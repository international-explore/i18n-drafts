UTC

UTC 也被称为GMT（Greenwich Mean Time 格林威治标准时间的缩写）。两者之间存在细微差异，但通常可以忽略。

UTC（Universal Coordinated Time 协调世界时的缩写）是现代计时的基础。除此之外，它还提供了增量时间 和 本地时间 之间的转换功能。

UTC的时区偏移量为0. UTC 通常使用Z以基于字段的格式。

基于字段的时间格式 - 时间string

当您使用基于字段的格式 编写时间时，您将日期与时间划分为独立的字段，比如年、月、日、时、分、秒等，形如2016-09-11T06:10:32. 与另一种非基于字段的表示方法时间方法对比，1465621816590表示同样的时间, 易读性比较差

基于字段的时间表示与 UTC或本地时间并没有强关联。 基于字段的时间通常也与特定的日历（例如公历）相关联。

ISO 8601 标准描述的格式是就是基于字段的。

增量时间 - 时间戳

增量时间是一种在计算机中表示时间的方式，它基于从特定时间点（称为“纪元”）单调增加的固定整数单位的进展。

Java（和许多其他系统）计时方式为，自 UTC1970年1月1日午夜（00:00 a.m.）以来的毫秒数（减去所有中间的闰秒）之和。其他系统使用不同的单位和纪元。

例如，在编写本段时，我的浏览器中 JavaScript 的增量时间是 1465621816590（即 2016 年 6 月 11 日上午 6 点 BST）。

大多数编程语言和操作环境提供或使用增量时间来解决时间问题。但是，用户通常不会看到增量时间,他通常转换为 基于字段的时间格式方便人们使用

时钟时间/当地时间

时钟时间 或者 本地时间 是一个时刻, 如果具备了时区信息后，则可以映射到增量时间的特定点上。但它对应于人们通过查看安装在特定地点墙上的时钟与日历时所识别的时间。

在HTML中local并不是指时钟时间 – 相反 描述为 datetime-local的时间实际上是一个浮动时间。

所以，例如，在写上一节的时候，我可能看了一眼我的电脑显示的时间，看到了6月11日星期六 06:10。如果应用时间和 UTC相关的知识 （在我的示例中，考虑到英国的夏令时，调整一小时），可以转为为增量时间1465621816590。

也可以转换为另一个区域的时钟时间, 比如在旧金山，假如有人同时看计算机的时间显示，就会看到6月10日星期五 22:10.

观测时间一词是指一种不同的方式来确认时间，他可以依赖于太阳，月亮或者星星等事物的物理观察，不过当前实际上不用。观测时间的中午是指太阳在空中的最高点，但实际上对于相距几公里的人来说，实际的中午时间会有所不同

浮动时间

部分时间值并不能映射到增量时间中的特定时刻。相反，它们代表一个名义时间，在世界各地的所有时区都以相同的方式描述。

举个例子，2016年6月11日星期六恰好是英国女王正式90岁的生日的日期。由于不同国家时钟设置的不同，6月11日在英国开始或结束的具体时间，在其他国家实际上有可能是周五或周日。但该事件的日期总是被称为6月11日星期六。

由于这种时间没有对应固定的增量时间，我们通常将此类时间称为浮动时间。当您将时区信息应用于浮动时间时，它们会产生 一系列可接受的增量时间值。

浮动时间不附加，也不应附加到特定时区。

浮动时间事件的其他示例包括报纸的出版日期、里约奥运会的开始日期、新年开始的时间、办公时间设置为“9 点到 5 点”而不考虑time zone等等。

时区

时区 是一组用于确定本地时间（时钟时间）的规则，它与特定地理区域的增量时间（在大多数计算系统中使用）有关，反之亦然。

时区规则必须考虑时区偏移增减 以及适用的挂钟时间的任何夏令时修改。

时区偏移量

时区偏移量 基于世界各地相对于本初子午线的位置，针对UTC时间做增减。通常偏移量以一小时为间隔，但偏移量也可以包括其他差异，例如 30 或 45 分钟。

在基于字段的时间格式中表示区域偏移的常用方法是，使用 +/- 后跟偏移。例如，日本比UTC时间早9小时，因此您可能会看到时间写为2016-06-11 05:10+09:00。

请注意，重要的是，区域偏移不会帮助您将时间转换为夏令时生效的时钟时间。

夏令时

夏令时 (DST) 或者“夏季时间”被采用，是为了让人们在晚上有更多日照时间。

DST 因国家/地区而异（更不用说因地区而异了），并且通常会进行特殊的一次性更改以适应特殊事件。并非所有地区都遵守夏令时：通常靠近赤道的地区不需要它。

在转换时间时，重要的是要了解当地何时引入 DST，有时会放弃 DST，以及 DST 开始和结束的日期（每年可能有所不同）

例如，韩国标准时间和日本标准时间目前使用相同的时区偏移量，也没有使用夏令时。 但是，日本在1951年放弃了DST，而韩国在1988年使用它，因此跟踪时间值的应用程序可能需要单独跟踪这些时区。

时区标识

时区标识允许您引用与 UTC的特定差异，包括 时区便宜 和夏令时。

识别时区规则集的最权威参考是 TZ database（也称为奥尔森时区数据库）, 它被各种商业 UNIX 操作系统、Linux、Java、CLDR、ICU 和许多其他系统和库等系统使用。但也存在其他系统：例如，Microsoft Windows 使用自己的数据集和标识符。

在 TZ 数据库中，时区的 ID 通常由地区和示例城市组成。示例城市是所讨论时区中的城市，应该为使用该时区的人所熟知。例如，美国太平洋时区的 TZ 数据库 ID 为America/Los_Angeles. TZ 数据库还为许多 ID 提供别名; 比如 Asia/Ulan Bator 等于 Asia/Ulaanbaatar.

Common Locale Data Repository (CLDR) 可用于为 ID 提供本地化形式。请注意，某些系统（例如 Apple 的 Mac OS）提供了其他示例城市。

公历和其他日历

公历 是使用最广泛的民间时间表示 。它是太阳历, 一年通常365天，增加了“闰年”的概念。 每4年增加一天，当年份可以被100整除不用增加（如果可以被400整除，需要增加）。

世界各地还有许多其他日历在使用，其中一些是阴历，一些基于与公历不同的开始日期，还有一些在每次名人去世时重置。通常这些日历仍用于宗教目的，但有时您也会发现它们被用于报纸和电子邮件，或用于出生日期。

ICU 或 Dojo 等技术支持不同日历系统之间的转换。
